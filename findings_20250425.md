## Findings

### P0 — Critical

1. **[src-tauri/tauri.conf.json:24] CSP is completely disabled**
   - `"csp": null` disables all Content Security Policy enforcement. For an email client that renders untrusted HTML from arbitrary senders, this is the single most dangerous misconfiguration: a malicious email can execute JavaScript, exfiltrate mailbox content, keylog keystrokes, or pivot to local resources.
   - **Suggested fix**: Set a restrictive CSP that at minimum blocks `script-src`, `object-src`, `frame-src` to `'none'` or `'self'`, and only allows `img-src data: https:` for inline CID images. The Reader already uses a sandboxed iframe — CSP is the defense-in-depth layer on top.

2. **[src-tauri/Cargo.toml:56-60] SQLCipher disabled — database stored unencrypted**
   - The `rusqlite` feature is `bundled` (not `bundled-sqlcipher-vendored-openssl`). The cipher key is hardcoded as `"__unencrypted_dev__"`. All email bodies, envelope metadata, and account server addresses sit in plaintext on disk at `crystalmail.sqlite`.
   - The TODO is well-documented, but this **must not ship to users**. A stolen laptop = full mailbox dump.
   - **Suggested fix**: Track the SQLCipher re-enablement as a release blocker. The keyring integration for the cipher key is already sketched in the comments.

### P1 — High

3. **[src-tauri/src/application/spam_rules.rs:82-95] Regex recompiled on every envelope during sync**
   - `matches()` calls `Regex::new(pat)` for every `SubjectRegex` rule on every envelope. During a sync of 500 messages with 5 regex rules, that's 2,500 regex compilations — each non-trivial.
   - **Suggested fix**: Compile regexes once per sync run (in `drive_uid_fetch`, after loading rules from DB) and pass a `HashMap<SpamRuleId, Regex>` alongside the rule list. The `validate_pattern` at save time already guarantees the pattern compiles.

4. **[src-tauri/tauri.conf.json] CSP null — (duplicate emphasis of #1)**
   - Covered above. This also affects the Tauri IPC boundary: without CSP, a compromised renderer can call arbitrary Tauri commands via the `__TAURI__` global.

### P2 — Medium

5. **[src-tauri/src/commands/mod.rs:41-68] `db_ping` writes to the database on every liveness probe**
   - Sends an `UpsertFolder` with a nil-uuid account to test the writer. The FK constraint prevents data corruption, but each call creates a failed write transaction. A frontend polling this during development adds unnecessary write load.
   - **Suggested fix**: Use a read-only probe (e.g., `PRAGMA user_version`) for the liveness check. The writer-alive check can be a dedicated `WriteCmd::Ping` that does nothing.

6. **[src-tauri/src/infrastructure/db.rs:350+] `dispatch` function is 200+ lines of match arms**
   - Each `WriteCmd` variant gets a 2-line arm that calls a function and sends the ack. This is mechanical but grows the file monotonically. At 30+ variants it's already hard to scan.
   - **Suggested fix**: Consider a macro or a trait-based dispatch table. Or at minimum, split the file into `db_writer.rs` (actor + dispatch) and `db_ops.rs` (the concrete SQL functions).

7. **[src-tauri/src/infrastructure/account_actor.rs] Account actor is fully stubbed**
   - `Sync`, `Archive`, and `Shutdown` all return `Err("not implemented")`. The comment says "This is the skeleton." If the actor pattern is the long-term plan, the current sync path (opening a new IMAP session per operation in `sync.rs`) is the interim solution — but there's no tracking issue or TODO linking them.
   - **Suggested fix**: Either remove the stub or add a `// TODO(#issue): integrate sync path through actor` comment so it doesn't become dead code.

8. **[src-tauri/src/commands/mail.rs:528-533] Duplicate `cancel_pending_fetch` call in `runOptimisticRemoval`**
   - The frontend `App.tsx` calls `invoke("cancel_pending_fetch", { messageId: id })` twice in the same callback (lines ~449 and ~465). The second call is a no-op (the first already removed the token), but it's dead code that suggests a copy-paste error.
   - **Suggested fix**: Remove the duplicate call.

9. **[No test files found] Zero test coverage**
   - No `#[test]` functions in any Rust file. No `*.test.ts` / `*.spec.ts` files. The `cargo test` command in AGENTS.md would run nothing.
   - **Suggested fix**: Start with unit tests for `spam_rules::matches`, `workflows::build_argv`, `sync::map_fetch`, and `queries::list_unified_folder`. These are pure-ish functions that are easy to test and high-value to protect.

10. **[src-tauri/src/infrastructure/queries.rs:197-228] SQL uses string-interpolated view names**
    - `list_unified_folder` builds SQL with `format!("... FROM {view} ...")`. The `view` variable is whitelist-controlled (safe today), but the pattern violates parameterized-query discipline. If someone adds a non-static entry to the whitelist, it's instant SQL injection.
    - **Suggested fix**: Add a `debug_assert!` or `const` array validation that the view name matches `^unified_\w+$` before interpolating. Or restructure to use a CTE/subquery approach that avoids interpolation entirely.

11. **[src-tauri/src/application/smtp.rs] No input validation on recipient addresses**
    - `SendMailRequest.to`, `.cc`, `.bcc` are passed directly to `lettre`'s `.parse()`. A malformed address returns a user-facing error, but there's no length limit or count limit. An attacker calling the Tauri command with 10,000 BCC addresses could cause memory pressure.
    - **Suggested fix**: Add a reasonable cap (e.g., 100 total recipients) at the command boundary.

### P3 — Low

12. **[src-tauri/src/state.rs] `AppState` has 10+ fields mixing unrelated concerns**
    - Pi config, spam analysis state, workflow config, prefetch tracking, and fetch cancellation all live in one struct. A `PiState` and `WorkflowState` sub-struct would improve clarity.

13. **[src-tauri/src/infrastructure/imap_client.rs] German-language log messages in `test_login_verbose`**
    - Messages like `"TCP-Verbindung zu"`, `"Authentifizierung erfolgreich"` are shown in the UI's verbose test report. If the app ever needs i18n on the Rust side, these will need extraction. Fine for now since the audience is German-speaking.

14. **[Cargo.toml] `keyring = "3"` with explicit backend features**
    - The comment warns about the in-memory mock fallback. The features are correctly set for Windows/macOS/Linux. Just flagging for awareness.

## Additional Suggestions

- **Add `#[cfg(test)]` modules** to the core Rust files. The `spam_rules`, `workflows`, and `queries` modules are the highest-value targets.
- **Pin the `pi` binary version** or add a version check at startup — the RPC protocol is internal and a mismatch could cause silent failures.
- **Consider a `deny.toml`** or `cargo-audit` in CI to catch CVEs in the dependency tree.
- **The `RunningGuard` pattern in `prefetch.rs`** is elegant — consider using the same pattern for the spam-analysis cancel state instead of the separate `AtomicBool`.
